1. Introduction

GAIT (GraphQL API Interactive Tool) is designed to bridge the gap between users and complex GraphQL APIs by allowing interaction through natural language constructs. The tool leverages NLP, voice recognition, and real-time API introspection.

GraphQL has become a widely adopted query language for APIs, offering flexibility and efficiency in data retrieval. However, interacting with GraphQL APIs requires technical knowledge of query structures, schema introspection, and API endpoints. This complexity presents a barrier for non-technical users and even slows down experienced developers when exploring new APIs. To address this challenge, GAIT (GraphQL API Interactive Tool) was developed as an intuitive web-based framework that allows users to interact with GraphQL APIs using natural language constructs. By leveraging Natural Language Processing (NLP), voice recognition, and real-time schema introspection, GAIT enables users to query GraphQL APIs without manually writing queries.

2. Related Work

Natural language interfaces for databases have been widely studied, primarily in the context of SQL-based systems. Many solutions, such as ColloQL and Seq2SQL, focus on translating user queries into structured SQL commands. These tools use rule-based approaches or deep learning models to understand user intent and generate queries accordingly. However, these systems are fundamentally tied to relational databases, which have predefined schemas. In contrast, GraphQL APIs require dynamic schema introspection, as API structures may change frequently. GAIT overcomes this limitation by integrating real-time introspection, allowing it to dynamically generate queries based on the latest API schema.

Conversational agents such as Google Dialogflow and IBM Watson Assistant have also explored natural language interaction with structured data. These tools allow developers to create predefined intents and responses for user queries, enabling some level of API interaction. However, they lack the flexibility to handle dynamic GraphQL schemas without extensive manual configuration. Unlike these systems, GAIT does not rely on predefined intents or static query mappings. Instead, it dynamically analyzes GraphQL schemas and generates valid queries in real-time, making it a more adaptable and scalable solution for API interaction.

Several tools and platforms exist for interacting with GraphQL APIs, but they often require technical expertise. Notable related work includes:

  • GraphiQL: A web-based interactive GraphQL IDE allowing developers to test and validate queries.
  • ColloQL: A system enabling natural language-based SQL queries, which serves as inspiration for GAIT's query processing pipeline.
  • SPARQL Query Builders: Various RDF-based query builders exist, but few integrate real-time NLP processing for dynamic query generation.
  • Voice-Driven Assistants: Platforms such as Google Assistant and Alexa provide voice-enabled querying, but their integration with GraphQL is minimal.

3. System Architecture

GAIT follows a service-oriented architecture consisting of:

  • Frontend: A React-based SPA for UI interaction.
  • Backend: A FastAPI service handling query processing.

    • The backend of GAIT is built on FastAPI, leveraging modern Python-based web frameworks to provide a high-performance API service. It consists of multiple components:

    • GraphQL Query Engine: Processes natural language inputs, maps them to GraphQL queries, and executes them against various APIs.
    • SPARQL Query Engine: Stores executed queries in RDF format and allows retrieval via SPARQL queries.
    • NLP Processing: Uses SpaCy and deep-translation to convert user inputs into structured queries.
    • Data Persistence: Queries and responses are stored in RDF format to ensure semantic interoperability.
  • APIs: Google Speech-to-Text, Google Translate and GraphQL endpoints.

The architecture of GAIT follows a service-oriented approach, designed to process natural language queries and interact with multiple GraphQL APIs efficiently. The system consists of several core components that work together to transform user inputs into structured GraphQL queries and return meaningful results.

The GAIT system is composed of three major layers: User Interface (UI), Processing Layer, and API Integration Layer.

  • User Interface (UI): Built as a React-based Single Page Application (SPA), accepting both text and voice input for querying GraphQL APIs and displaying formatted responses.
  • Processing Layer: Includes:
    • Natural Language Understanding (NLU) Module: Processes user queries, extracts intent, and translates them into structured formats.
    • GraphQL Introspection Service: Analyzes and retrieves the schema of target GraphQL APIs.
    • Query Generator: Constructs valid GraphQL queries.
    • Result Formatter: Formats API responses into a user-friendly format.
  • API Integration Layer: Manages requests to external GraphQL APIs, ensuring efficient data retrieval and error handling.

The GAIT system follows a structured workflow for query processing:

  • User Input Processing: Users submit queries via text or voice. If voice input is used, it is converted to text using the Google Speech-to-Text API.
  • Query Generation: The NLU Module extracts key information, GraphQL Introspection Service retrieves API schema details, and the Query Generator constructs a valid GraphQL query.
  • Data Retrieval & Response Formatting: The API Integrator sends the query to the GraphQL API, retrieves results, and the Result Formatter processes and displays them to the user.

GAIT’s architecture provides several advantages:

  • Modular and Scalable: Supports easy integration with new GraphQL APIs and NLP improvements.
  • Flexible Input Handling: Enables both text and voice-based interactions for a seamless experience.
  • Efficient Query Processing: The introspection service dynamically constructs accurate GraphQL queries.

By following this architecture, GAIT provides an intuitive, intelligent, and flexible way for users to interact with GraphQL APIs using natural language.

4. Backend Query Services

The backend services of GAIT handle various aspects of GraphQL query processing, SPARQL data management, and multilingual support. It is designed to efficiently manage and process user queries, convert natural language into structured API requests, and store these interactions for future reference.

4.1 GraphQL Query Processing

GAIT enables users to interact with multiple GraphQL APIs by converting natural language queries into structured GraphQL requests. The backend follows these steps:

  • Query Interpretation: User input is processed using NLP models to extract key entities.
  • API Mapping: The system identifies the relevant GraphQL API based on extracted entities.
  • Query Construction: Dynamic GraphQL queries are generated based on user intent.
  • Execution & Response: Queries are executed, and the JSON response is structured for readability.

This approach allows seamless interaction with multiple GraphQL APIs, providing users with accurate and real-time results.

4.2 SPARQL Query Storage

GAIT incorporates an RDF-based query storage system using SPARQL, which allows structured representation of user queries. The benefits of this approach include:

  • Permanent storage of user queries with metadata.
  • Efficient retrieval of past queries using SPARQL queries.
  • Integration with Linked Data applications for extended interoperability.

The system ensures that user interactions are logged in a format that supports efficient querying and analysis of API usage trends.

4.3 Multilingual Query Support

GAIT provides multilingual query support, making it accessible to users in different linguistic regions. This feature is implemented through:

  • Language Detection: The backend automatically detects the language of the query using DeepL.
  • Translation: If the detected language is not English, the query is translated into English before processing.
  • Response Handling: Query results are translated back into the user’s preferred language.

Currently, GAIT partially supports Romanian, German, and English, with future plans to extend support for additional languages.

4.4 Backend Infrastructure & Performance Optimization

GAIT’s backend is built on FastAPI, ensuring high-performance processing of API queries. Several optimizations have been implemented:

  • Asynchronous Processing: FastAPI’s async capabilities allow non-blocking query execution.
  • Caching: Frequently accessed API responses are cached to reduce redundant requests.
  • Load Balancing: Requests are distributed efficiently to ensure system stability under high loads.
  • Database Optimization: The RDF store is indexed to allow fast retrieval of stored queries.

These enhancements enable GAIT to handle a high volume of user interactions with minimal response time.

5. Implementation

The tool consists of a modular backend using FastAPI and GraphQL processing via gql. The API integrator supports GraphQL, SPARQL, and RDF query handling.

The backend services handle natural language query conversion, API interaction, and RDF storage. The key implementations include:

  • Query Parsing: The NLP engine processes user queries, extracts entities, and maps them to GraphQL-compatible structures.
  • GraphQL Execution: The parsed queries are sent to public GraphQL APIs (e.g., Country API, AniList API) using HTTP clients.
  • SPARQL Query Storage: Every query and its response are stored in an RDF triple store, allowing structured queries via SPARQL.
  • Multi-Language Support: Uses translation APIs to support multiple languages, dynamically translating queries to English.

The Query Generator API is responsible for generating GraphQL queries based on natural language input. It processes user queries, maps them to appropriate GraphQL APIs, and returns structured data.

Frontend Implementation

The frontend of GAIT is built using React, a JavaScript library for developing interactive user interfaces. It focuses on providing a seamless and user-friendly experience by integrating voice input, multi-lingual support, and authentication mechanisms.

App Component

The App component acts as the core of the frontend application. It employs React Router for navigation and wraps the entire app within the UserContextProvider to manage user authentication and state. 

import React from "react";
              import { BrowserRouter, Routes, Route } from "react-router-dom";
              import QueryPage from "./pages/QueryPage";
              import Layout from "./components/Layout";
              import HomePage from "./pages/HomePage";
              import TutorialPage from "./pages/TutorialPage";
              import UserContextProvider from "./hooks/context";
              import UserProfile from "./pages/UserProfile";
              import SignInPage from "./pages/SignInPage";
              import NotFound from "./pages/NotFound";
              import PrivateRoutes from "./components/PrivateRoutes";
              import ResultsPage from "./pages/ResultsPage";
              
              function App() {
                return (
                  <div className="App">
                    <BrowserRouter>
                      <UserContextProvider>
                        <Routes>
                          <Route path="*" element={<NotFound />} />
                          <Route path="/" element={<Layout />}>
                            <Route index element={<HomePage />} />
                            <Route path="tutorial" element={<TutorialPage />} />
                            <Route path="sign-in" element={<SignInPage />} />
                            <Route element={<PrivateRoutes />}>
                              <Route path="query" element={<QueryPage />} />
                              <Route path="profile" element={<UserProfile />} />
                              <Route path="result" element={<ResultsPage />} />
                            </Route>
                          </Route>
                        </Routes>
                      </UserContextProvider>
                    </BrowserRouter>
                  </div>
                );
              }
              
              export default App;

Authentication and Private Routes

The PrivateRoutes component ensures that only authenticated users can access specific pages. It checks authentication status via the useAuth hook and redirects unauthorized users to the sign-in page. 

import { Outlet, Navigate } from "react-router-dom";
          import { useAuth } from "../hooks/context.js";
          
          const PrivateRoutes = () => {
            const { userData } = useAuth();
            return userData.email !== "" ? <Outlet /> : <Navigate to="/sign-in" />;
          };
          export default PrivateRoutes;

Layout and User Interface

The Layout component defines the main structure of the application, including navigation menus that dynamically adjust based on user authentication status. This ensures a fluid user experience, whether the user is signed in or browsing anonymously.

User Context and Authentication

The UserContextProvider manages authentication state, login, and registration processes. It also utilizes localStorage to persist session data across page reloads.

Query Submission

The QueryPage component is the primary interface for users to interact with GraphQL APIs. It supports both text-based and voice-based query inputs, utilizing the webkitSpeechRecognition API for speech-to-text conversion. Upon submission, queries are processed and sent to the backend for execution.

Sign-In and Authentication Handling

The SignInPage handles user authentication by validating email and password credentials. It integrates form validation techniques and manages authentication status via the useAuth hook.

User Profile Management

The UserProfile component allows users to update their profile, manage their query history, and modify their settings. It provides functionality for changing passwords, uploading profile pictures, and managing saved queries.

Diagram Description

The process begins with user authentication, where login credentials are verified against the User Profile Database. Once authenticated, the user submits a query via voice input, which is processed using the Google Speech-to-Text API to convert audio into text. If necessary, the text is then sent to the Google Translate API for language conversion. The translated or original query is forwarded to the Query Generator API, which constructs a valid GraphQL request. This request is passed to the Integrator API, which communicates with public GraphQL APIs to fetch the required data. If translation is required, the results are sent back to the Google Translate API before being stored in the Query History Database and displayed to the user. Finally, users have the option to bookmark or download results, with actions being managed by the Query Generator API and stored in the system database.

Workflow Summary

The GAIT system follows a structured workflow that begins with user authentication. Once logged in, users submit queries either through voice or text input. The system processes the input, applying translation if needed, and generates a valid GraphQL query. This query is executed against external GraphQL APIs, retrieving relevant data. If required, results are translated back into the user’s preferred language before being displayed. Additionally, users have the option to bookmark or download their query results for future use, enhancing usability and accessibility.

🏗️ API Endpoints

🔹 Process Natural Language Query

Endpoint: POST /query

Description: Converts a natural language query into a GraphQL query and returns results.

Request Body:

{
  "user_input": "What is the capital of France?"
}

Response:

{
  "query": "{ country(code: \"FR\") { name capital currency languages { name } } }",
  "response": {
    "data": {
      "country": {
        "name": "France",
        "capital": "Paris",
        "currency": "EUR",
        "languages": [{ "name": "French" }]
      }
    }
  },
  "link": "http://localhost:5001/query/8a056e1a-3897-4440-a391-a3d291c1de8c"
}

🔹 Retrieve Stored Queries

Endpoint: GET /queries

Description: Returns a list of stored query IDs.

Response:

[
  { "queryID": "http://localhost:5001/query/8a056e1a-3897-4440-a391-a3d291c1de8c" },
  { "queryID": "http://localhost:5001/query/3faa5b9b-8f81-401b-b5dd-8955fbeda615" }
]

🔹 Execute SPARQL Query

Endpoint: POST /sparql

Description: Executes a SPARQL query on stored RDF data.

Request Body:

{
  "query": "PREFIX ex: <http://example.org/queries#> SELECT ?userQuery WHERE { ?query ex:userQuery ?userQuery . }"
}

Response:

[
  { "userQuery": "What is the capital of France?" },
  { "userQuery": "Capital of Germany?" }
]

7. Challenges & Solutions

During the development of GAIT, several challenges were encountered, requiring innovative solutions to ensure smooth functionality and user experience.

Natural Language Understanding Accuracy

One of the primary difficulties was ensuring that the system correctly interprets natural language queries and maps them to valid GraphQL queries. Misinterpretation of user intent could lead to incorrect or failed API requests.

Solution: To address this, we integrated a Natural Language Understanding (NLU) module that leverages machine learning models trained on diverse datasets.

Voice Input Processing

Recognizing and converting voice queries into structured text while maintaining accuracy was a significant challenge, especially with variations in accents and speech patterns.

Solution: We utilized the Google Speech-to-Text API, which provides high accuracy in voice recognition.

GraphQL Query Generation

Dynamically constructing correct GraphQL queries based on user intent and available API schema required careful handling to avoid syntax errors and missing parameters.

Solution: The Query Generator was designed to use schema introspection, ensuring that only valid API queries are formed.

Efficient API Integration and Response Formatting

Different GraphQL APIs have varied response structures, making it challenging to standardize results for the user interface.

Solution: We developed different templates for the results page in the User Interface for every public API recognized by the Query Generator.

8. Future Work & Conclusion

While GAIT provides a solid foundation for natural language interaction with GraphQL APIs, several areas for future improvements have been identified. These enhancements aim to make the platform more robust, user-friendly, and capable of handling a wider range of use cases.

Enhanced NLP Capabilities

To improve accuracy, we aim to implement state-of-the-art machine learning models, such as transformer-based architectures (e.g., BERT, GPT), to enhance intent recognition. Additionally, entity extraction will be refined to better identify key elements like dates, names, and locations within user queries.

Integration with More GraphQL APIs

We plan to expand the range of supported GraphQL APIs, including public APIs such as GitHub and Shopify, as well as allowing users to configure their own API endpoints dynamically. Furthermore, an API discovery feature will be introduced to help users explore and integrate APIs seamlessly.

Performance Optimization

Optimizing query execution speed will be a key focus, including implementing parallelized API requests, refining the NLU module, and leveraging caching mechanisms to reduce redundant API calls.

Improved User Experience

Enhancing error handling by providing more informative messages and troubleshooting guides will improve usability. Additionally, personalization features, such as user preferences for language and theme settings, will be introduced.

Security and Privacy Enhancements

To protect user data, we will implement end-to-end encryption for sensitive information, such as user credentials and API keys.

Integration with Other Technologies

Future developments will include integrating GAIT with chatbot platforms like Slack and Microsoft Teams, as well as extending voice command support to smart assistants like Amazon Alexa and Google Assistant.
